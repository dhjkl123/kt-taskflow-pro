## 왜 이렇게 나누는가? (관심사 분리 원칙)

이 문서 체계는 **관심사 분리(Separation of Concerns)** 원칙을 따릅니다.

### 각 단계별 목적

#### **1단계: 개요 (00-overview)**
- **관심사**: 문서 네비게이션
- **대상**: 모든 개발자
- **목적**: "어떤 문서를 읽어야 하는가?"를 명확히 함

#### **2단계: 제품 (01-product)**
- **관심사**: 비즈니스 요구사항
- **대상**: PM, 디자이너, 풀스택 개발자
- **목적**: "이 프로젝트는 무엇인가?" (기능, 사용자, 목표)

#### **3단계: 기술명세 (02-specs)**
- **관심사**: 기술적 구현 세부사항
- **대상**: 백엔드/프론트엔드 개발자
- **목적**: "어떤 기술을 사용하는가?" (스택, API, DB)

#### **4단계: 설계 (03-design)**
- **관심사**: 시스템 아키텍처
- **대상**: 시니어 개발자, 아키텍트
- **목적**: "어떻게 만들 것인가?" (구조, 의존성, 설계 결정)

#### **5단계: 작업 (04-tasks)**
- **관심사**: 구현 체크리스트
- **대상**: 모든 개발자
- **목적**: "뭘 만들어야 하는가?" (기능 목록, 우선순위)

#### **6단계: 규칙 (05-conventions)**
- **관심사**: 코드 품질 기준
- **대상**: 모든 개발자
- **목적**: "어떻게 작성해야 하는가?" (스타일, 표준)

### 왜 이 순서인가?

```
비즈니스 요구사항 → 기술 선택 → 아키텍처 설계 → 구현 작업 → 코드 품질
    (01)        →   (02)   →    (03)      →   (04)  →   (05)
    
                         ↑
                    00에서 네비게이션
```

1. **먼저 "무엇"을 알아야** (제품, 기술)
2. **그 다음 "어떻게"를 설계하고** (아키텍처)
3. **마지막에 "어떤 기준으로"를 정한다** (규칙, 컨벤션)

이렇게 하면:
- 🎯 **명확한 의사결정**: 각 단계가 이전 단계의 결과 위에 구축됨
- 🔄 **유지보수 용이**: 변경 사항이 발생할 때 어디를 수정할지 명확함
- 👥 **역할별 관심사 명확**: PM은 01, 개발자는 02-05에 집중
- 📚 **온보딩 가속화**: 새로운 팀원이 순서대로 읽으면 자연스럽게 이해
